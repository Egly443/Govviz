# Govviz

Long-run performance data for UK government departments, presented in context
rather than in isolation. Each department has a hero trend, a set of core
"competence" measures, optional supporting context, and a per-indicator RAG
snapshot (deliberately **not** reduced to a single league-table grade).

Built with **Vite + React + TypeScript**, **TanStack Router** (file-based
routing), **Tailwind CSS v4**, **Recharts**, and **lucide-react**.

## Departments

HMT (Treasury), DHSC (Health & Social Care), DfE (Education), Home Office,
MoJ (Justice), MoD (Defence), DWP (Work & Pensions), DfT (Transport),
MHCLG (Housing, Communities & Local Government), and Defra (Environment,
Food & Rural Affairs).

> **Real data only.** Every chart shows real data fetched from official sources
> (ONS, World Bank, gov.uk, NHS England, the Environment Agency, DfE EES and
> others) in CI and baked into the production build. There is no fabricated or
> illustrative data: a series with no wired source renders an explicit "no
> source yet" placeholder rather than an invented trend. See `sourceUrl` on each
> series and `scripts/build-data.mjs` for the per-series sourcing, and
> `docs/conformance/` for the AI-readiness conformance suite and series profile.

## Open data — the essay, made real

Govviz re-publishes every series as a **reference implementation of its own
[AI-ready series profile](docs/conformance/ai-ready-series-profile.md)** — the
machine-first publishing standard argued for in
[the essay](docs/blog-open-data-for-ai.md). Built into `dist/data/` and served at
[`/Govviz/data/`](https://egly443.github.io/Govviz/data/):

| Current status | Link / source |
|---|---|
| Live data portal | [`/Govviz/data/`](https://egly443.github.io/Govviz/data/) |
| Generated series count and build date | [`series/index.json`](https://egly443.github.io/Govviz/data/series/index.json) (`count` and `built` are generated by `scripts/build-open-data.mjs`) |
| Conformance profile | [`profile.json`](https://egly443.github.io/Govviz/data/profile.json) / [`ai-ready-series.schema.json`](https://egly443.github.io/Govviz/data/ai-ready-series.schema.json) |
| Catalogue and graph | [`catalog.json`](https://egly443.github.io/Govviz/data/catalog.json) / [`graph.jsonld`](https://egly443.github.io/Govviz/data/graph.jsonld) |
| Agent/static contracts | [`mcp.json`](https://egly443.github.io/Govviz/data/mcp.json) / [`openapi.json`](https://egly443.github.io/Govviz/data/openapi.json) |
| Observability | [`health-history.json`](https://egly443.github.io/Govviz/data/health-history.json) / [`conformance-report.html`](https://egly443.github.io/Govviz/data/conformance-report.html) |
| Benchmark suite | [`docs/benchmarks/`](docs/benchmarks/) |

- **Stable, resolvable id per series** → `GET /data/series/{id}.json` returns the
  metadata record: unit (+ SDMX multiplier), coverage, periodicity, revision
  status, provenance with verifiable upstream lineage, licence, next release, and
  a published **`validRange`** a consumer uses to reject a wrong-but-plausible value.
- **Tidy data** → `/data/series/{id}/data.csv` (long-format, one observation per
  row) with a **CSVW** schema and a content-versioned twin beside the `latest` alias.
- **DCAT catalogue** (`/data/catalog.json`) and an open-agent **MCP** interface
  (`tools/mcp/govviz-mcp.mjs`).
- **Self-certifying:** `scripts/check-open-data.mjs` fails the build if any record
  is non-conformant or any value falls outside its `validRange`.

So an agent — or a fifteen-line script — can read any series with
`resolve id → GET tidy data`, no scraping. Govviz is a downstream compiler; every
record names its primary producer. Licence: see [`DATA-LICENCE.md`](DATA-LICENCE.md).

## Non-goals

Govviz does not centralise primary government data, replace ONS/GSS methodology,
publish restricted microdata, bypass disclosure control, or certify official
statistics. It demonstrates a thin publication profile, downstream compiler,
and conformance harness for already-published public aggregates.

## Getting started

```bash
npm install
npm run dev      # start the dev server (http://localhost:5173)
npm run build    # production build to dist/
npm run preview  # preview the production build
npm run typecheck
```

> A local build without CI-baked data shows placeholders for every series
> (illustrative fallbacks were removed by design). The live site is fully
> populated because CI fetches the real data on deploy.

## Local real-data builds

The production deploy fetches live official-source data before building. To
reproduce one or more real-data series locally, install the optional parser
packages used by workbook/zip-backed sources, then run the fetcher:

```bash
npm install --no-save xlsx fflate
npm run fetch-data -- --only=defra-sewage-hours
npm run fetch-data -- --only=hmrc-call-wait,cab-foi-intime,desnz-electricity-price
```

Expect live network access and source-dependent runtime; broad fetches can take
many minutes because they follow official pages, collections, workbooks and
archives. The fetcher writes `src/generated/seriesData.ts`. That file is kept
empty in git and populated by CI, so restore it to the committed empty form
after local testing unless you are intentionally inspecting generated data.

## Reading & sharing the essay

The *Agentic Open Data* essay is published as a real page at **`/blog`**
(`egly443.github.io/Govviz/blog`) — not just a Markdown file — so it can be
measured and shared.

- **Analytics (optional, cookieless):** set a repo/Actions **variable**
  `VITE_GOATCOUNTER` to your GoatCounter count URL
  (`https://YOURCODE.goatcounter.com/count`) and the deploy build injects a
  privacy-first beacon (no cookies, no personal data → no consent banner
  required). Leave it unset and **no tracker is shipped**. It reports counts,
  referrers, countries and UTM campaigns — not individuals.
- **Sharing / traction:** post with the **UTM-tagged links** in
  [`docs/share-links.md`](docs/share-links.md) so you can see which channel — or
  which specific outreach (the ODI vs GDS email) — drove visits.
- **What you can/can't know:** how many, from where, which channel/recipient,
  which country — yes; named individual readers — no, unless they self-identify
  (an email-capture CTA) or click a link you personally sent them.

## Overview treemap

`/overview` is the landing page: a whole-of-government heatmap. Each
**department block is sized by its spending** (`spendBn`), and within a block
each indicator tile is **weighted Finviz-style** (hero > core > supporting).
Tile **colour** is a red→amber→green RAG score: where a published target/standard
exists, green means at or beyond that benchmark. Indicators with **no external
benchmark** are scored against their own history and shown **desaturated** (and
labelled in the legend) so an own-range score never reads as a verdict. Unsourced
indicators are greyed with no value. Clicking a tile opens its full `TrendPanel`
chart in a modal. A dot marks indicators that have an official target.

## Routing

`/` redirects to `/overview`. Each department is reachable at `/<code>`, e.g.
`/hmt`, `/dhsc`, `/dfe`, `/home-office`, `/moj`, `/mod`, `/dwp`, `/dft`,
`/mhclg`, `/defra`. Unknown codes render a 404 with a link back to DHSC.
